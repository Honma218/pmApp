# CLAUDE.md

このファイルは Claude Code がこのリポジトリで作業する際の前提・規約・守るべき原則を定義する。
詳細な要件は `docs/spec.md`（要件定義／設計仕様書）を参照すること。

## プロジェクト概要

客先常駐スタッフの業務報告（テキスト／音声）を蓄積し、AIで要約・専門用語化して
職務経歴書（スキルシート, Excel）を自動生成するWebシステム。

このシステムは4つの役割を持つ:
1. スタッフステータス管理
2. スキルシート生成
3. 業務報告の自動要約
4. スタッフ支援のためのコミュニケーション／マネジメント（ケアの入口）

## 技術スタック

- フロントエンド: Next.js (React) + TypeScript、レスポンシブ（スタッフはモバイルファースト、管理者はPC前提）
- バックエンド: Python / FastAPI
- DB: PostgreSQL（マイグレーションで管理）
- ファイル格納: S3互換オブジェクトストレージ（生成Excel・テンプレート・音声）
- 非同期/ジョブ: Celery + Redis または APScheduler（通知のスタッフ別時刻判定）
- AI要約: 提供元非依存の抽象化層の背後で呼ぶ（Claude / Gemini API / Vertex AI を差し替え可能）
- 音声テキスト化(STT): Web Speech API またはサーバ側STT（後フェーズ）
- Excel生成: openpyxl

## 開発コマンド（プレースホルダ：実際の値に更新すること）

- backend 起動: `cd backend && uvicorn app.main:app --reload`
- frontend 起動: `cd frontend && npm run dev`
- マイグレーション: `cd backend && alembic upgrade head`（新規: `alembic revision --autogenerate -m "..."`）
- テスト: `cd backend && pytest` / `cd frontend && npm test`

## アーキテクチャの重要原則（必ず守る）

これらは仕様の核であり、逸脱しないこと。曖昧なときは実装を止めて確認する。

1. **権限境界はバックエンドで強制する。** スタッフは自分のデータのみ、管理者は担当グループのみ。
   フロントの出し分けだけに依存しない。全エンドポイントで認可チェックを行う。
2. **AI要約は提供元非依存の抽象化層の背後でのみ呼ぶ。** 具体プロバイダ（Claude/Gemini/Vertex）に
   直接依存するコードをドメイン層に書かない。1つのインターフェースで差し替え可能にする。
3. **AIに数値・事実を創作させない。** 職務経歴の変換ではマスターに無い数値を出させない。
   不足は空欄のまま。AI出力はアンカー/フィールド対応のJSONスキーマに固定する。
4. **音声は「STT→要約」の2段。** AIの要約モデルに音声を直接文字起こしさせない。
   STTで得たテキストは確定前にユーザーが修正できること。
5. **Excelはアンカー方式 + openpyxl。** テンプレートのプレースホルダ/名前付き範囲に書き込み、
   繰り返しブロックは雛形行を複製して書式（罫線・結合・フォント）を維持する。データ取得と書き込みは決定的に。
6. **突合は増分マージ。** 全履歴を毎回再処理しない。案件キー単位でマージし、ステータス等は最新で上書き。
   生報告ログ(REPORTS)は確定後不変。
7. **マルチグループは設定駆動。** 職種/案件属性はデータ（グループ設定）で切り替える。
   グループごとにコード分岐を増やさない。
8. **雑感（メンタル面）は隔離する。** AI変換・スキルシート反映・一般管理者の閲覧から完全に除外。
   閲覧はHR/担当者と本人に限定。スコア化・ダッシュボード化しない。
9. **タイムゾーン**は保存UTC・表示ローカルで統一する。
10. **シークレット**（OAuth資格情報・APIキー）はコミットしない。環境変数で扱う。

## コーディング規約

- 型を付ける（TypeScript / Pydantic）。公開APIの入出力はスキーマで定義。
- ドメインロジックは services 層に集約し、ルーティングは薄く保つ。
- 変更は縦の薄いスライス単位でコミットし、コミットメッセージに意図を書く。
- 破壊的な操作（マイグレーション・削除）は事前に説明し、確認を取る。
