# 統合役の再検証は「当該スライス＋秘密＋差分」。全スイートとリグレッション検出は CI が持つ

付録B の未決「統合役再検証のスコープ（全スイートか当該スライスのみか）」を、**検出したいものが2つあり別物である**として解く。

| 検出したいもの | 担い手 |
|---|---|
| 「下流の環境でだけ緑だった」＝再現性の欠如 | **人（統合役）**。別マシンで走らせることに意味がある |
| 「他のスライスを壊した」＝リグレッション | **機械（CI）**。全 PR で常に、疲れずに回る |

したがって統合役はローカルで**当該スライスのスイートのみ**を再実行し、あわせて秘密チェックと差分の目視を行う。全スイートは CI が全 PR で回す。統合役は関所の番人であってテストランナーではない。網羅は機械に任せる。

## 必須の付帯条件

GitHub のブランチ保護で **「main が更新されたら再度 CI を通す」**（*Require branches to be up to date before merging*）を有効にする。これが無いと、**個別には緑な2つの PR がマージ後に赤になる**。統合役がどれだけ丁寧にローカル実行しても防げない——ローカルには他人の PR が存在せず、構造的に検出不能だからである。

## 帰結

CI で `reference-mock`（Python）・`backend`（Node）・`frontend`（Node）・Playwright ブラウザを起動できる必要がある。golden 比較のためブラウザ版を pin する（ADR-0008）。これは小さくないインフラであり、初スプリントで組まないと初期スライスはリグレッション検出なしで main に入る。

## 導入時期（決定）

**初スプリントでは CI を組まない。** 最初の数スライスは統合役が全スイートをローカル実行して代替する。目的C（初スプリント内に下流全員が1回完走）を CI 構築で遅らせないため、かつ段階導入の原則「効果最大・コスト最小から」に従うため。

**ただし期限を切る**——期限のない技術的負債は返済されない。

- **`docs/slices/README.md` のスライスが5本たまった時点**、または
- **初スプリント終了時の骨格見直し**

のいずれか早い方で CI を組む。Flywheel の観察項目に載せる。

この期間中、**「個別に緑な2つの PR がマージ後に赤」は構造的に検出できない**。統合役の落ち度ではないことを事前に全員へ伝える。
