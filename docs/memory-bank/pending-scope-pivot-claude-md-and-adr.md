# スコープ変更（業務アプリ再実装 → スライス進捗集計アプリのみ）に伴う憲法・ADR 改訂案

## 背景

本プロジェクトは当初「既存の業務アプリ（Repo Base／通称 staff-report-system）を参照仕様として
TypeScript(Express) → 将来 Java/Spring へ再実装する」ことを目的としていた（CLAUDE.md §0、
CONTEXT.md 冒頭、`M-team開発計画書_v2.md` 全体）。

2026-07-11 のセッションで、ユーザー（PM 相当）から「スライス進捗集計アプリのみの開発。
業務報告・スキルシート生成システムの記載は全面削除」との指示があった。
`スライス進捗集計アプリ` は元々 `tools/slice-aggregator/` に置かれた Harness-Keeper 向けの
KPI・進捗集計ツールで、その実装ロードマップ自身が「**これは MVP 本体ではない**」と明記していたが、
本指示によりこちらが唯一の開発対象になった。

`CLAUDE.md` と `docs/adr/**` は hook・permissions で AI の Edit/Write が構造的にブロックされている
（PM 専用）ため、このセッションでは以下の**改訂案を文書化するだけ**で、本体には反映していない。
README.md・CONTEXT.md・`docs/playbook.md`・`M-team開発計画書_v2.md` には、この場で編集可能な範囲で
移行中の注記を追加済み（削除ではなく「旧スコープにつき現行未使用」という注記に留めている。理由は
下記「編集方針」参照）。

## 編集方針（なぜ「削除」ではなく「注記」にしたか）

- `README.md`・`CONTEXT.md` は文字通りの記述（プロジェクト概要・用語定義）を新スコープ用に書き換えた。
- `docs/playbook.md`・`M-team開発計画書_v2.md` は分量が大きく（427行・888行）、業務アプリ固有の記述が
  全体に深く織り込まれている（工程3・4の受け入れテスト翻訳フロー、Express/Java移行の技術選定など）。
  内容を丸ごと削除すると、汎用的に再利用できる部分（層境ゲート・役割分担・モデル配分・Flywheel の
  仕組み）まで失われるため、**削除ではなく「現行スコープでは未使用」という注記を先頭に追加**した。
  完全な書き直し（スライス進捗集計アプリ専用の新しい工程表への置き換え）は別タスクとして
  PM の判断を仰ぐ（「未確認・次のアクション候補」参照）。

## CLAUDE.md 改訂案

### §0（このリポジトリは何か）

現行:
> Repo Base（`staff-report-system`／参照モック＝answer key）の挙動を、AI駆動のチーム開発で
> TypeScript(Express) + Next.js に再実装するプロジェクト。最終的に Java/Spring へ一括移行する。
> 第一目的は**再現可能な型（方法論）の確立**であり、機能前進ではない。

改訂案:
> **スライス進捗集計アプリ**（M-team 自身の AI駆動チーム開発の進捗・KPI を Git への投影として
> 集計するツール。`tools/slice-aggregator/`）を、AI駆動のチーム開発で開発するプロジェクト。
> 第一目的は**再現可能な型（方法論）の確立**であり、機能前進ではない（変更なし）。
> 実装フェーズの正本は `2026-07-11_slice-progress-aggregator_実装ロードマップ.md`（P0〜P6）。

### §1（絶対禁止）の該当箇所

- 「3. `feature/*` から `acceptance/` と `reference-mock/` を変更しない」「4. `spec/*` から
  `backend/` `frontend/` を変更しない」は、現行スコープにこれらのディレクトリが存在しない前提と
  なるため、削除するか「該当ディレクトリが復活した場合に適用」という条件付きに改める必要がある。
  ただし `docs/metrics/events/**` への bot 直コミット例外（ADR-0012）との整合は要確認。

### §5（アーキテクチャの骨格）

`backend/`・`frontend/`・`acceptance/`・`reference-mock/` の説明を削除し、`tools/slice-aggregator/`
（NDJSON イベント・fold 純関数・GitHub Actions 投影）を骨格として記述し直す必要がある。

## ADR 群の要改訂・廃止判定

`docs/adr/` は差し戻し・廃止を Harness-Keeper／PM が正式に決める領域のため、以下は**分類案**に留める。

### 廃止・全面改訂が必要（業務アプリの受け入れテスト＝reference-mock 前提で成立していたため）

| ADR | 主題 | 廃止理由 |
|---|---|---|
| 0001 | 受け入れテストはブラックボックス専用 | 「Java移行の完全仕様」「FastAPIモック（answer key）に先に流す」が前提。対象の業務アプリが無くなり前提が消失 |
| 0004 | `acceptance/` の書込権をブランチ名で判定 | 0001 前提の運用ルール。`acceptance/` 自体が現行スコープで不要 |
| 0005 | 参照モックを `reference-mock/` として vendor | 本セッションで vendor 済みの `reference-mock/` を全削除した（ユーザー承認済み）。ADR の対象が消滅 |
| 0008 | golden スクリーンショットで画面一致を機械判定 | 0005 前提（参照モックの画面と比較）。比較対象が無い |
| 0009 | 上流の一本道（`/spec` `/brief`）とブランチ・工程の対応 | `/spec` は「仕様表→受け入れテスト翻訳」が前提。現行スコープでは同名コマンドの意味が変わる |
| 0010 | 統合役の再検証スコープ（全スイート vs 当該スライス） | 「スイート」＝受け入れテストが前提。現行スコープでの再検証対象を再定義する必要がある |
| 0011 | Express＋構造規約（`router→service→repository`） | Java/Spring 移行を見据えたバックエンド実装の話。スライス進捗集計アプリ v0 は「サーバー/DBなし」（実装ロードマップ§0）につき無関係 |
| 0002 | NestJS 採用（0011 に supersede 済み） | 既に superseded。0011 が廃止なら参照元として歴史記録のみ残す |

### 現行スコープでも有効（業務アプリの具体的な中身に依存しない、汎用的な運用ルール）

| ADR | 主題 | 有効な理由 |
|---|---|---|
| 0003 | Summarizer が報告本文を第三者AIへ送るデータフロー | ※業務アプリ固有（「報告本文」＝業務アプリのデータ）。**現行スコープでは不要**（上表に含めるべきだったが対象データの性質が異なるため個別確認を推奨） |
| 0006 | スライス指示書の正本は repo のファイル（issue はポインタ） | 「指示書」の概念自体は何を作るかに依存しない汎用ルール |
| 0007 | 層境ゲートは全PRに掛け、重さを2段に分ける（PM判定） | ゲート構造は汎用。ただし `irreversible` ラベルのトリガーパス（`acceptance/**` 等）は要更新 |
| 0012 | metrics events への bot 直コミット許可 | スライス進捗集計アプリそのものの決定。現行スコープの中核として維持・優先実装 |

> 0003 は要再判定：スライス進捗集計アプリが将来 AI 要約等でサードパーティ AI へデータを渡す設計に
> なるなら別内容の ADR として書き直し、ならないなら廃止。
