# 層境ゲート（PMのGO/NO-GO）を `/gate` で記録してからマージする、という手順が宣言すらされていない

## 観察

slice-01 のドッグフーディング（PR #19）で、`/gate GO --slice 1` を一度も投稿せずに
`gh pr merge 19 --merge` で直接マージした。結果、`gate` イベントが一度も記録されないまま
`completed` になった。

`docs/adr/0007`（層境ゲート）・`docs/playbook.md`「スライス単位の関所」節は
「判定者はPM」「層境ゲートの記録」を求めているが、**「マージの前に `/gate` コマンドで記録する」
という手順そのものがどこにも書かれていない**。`/gate`・`/reject`・`/abandon`
（`.github/workflows/gate.yml`）は実装として正しく動作する（PR #15・#19 で確認済み）ため、
これは実装のバグではなく**手順の欠落**。

## 分類

ハーネスの欠陥（手順の欠落。playbook のどこにも「マージ前に `/gate` を打つ」ことが宣言されていない）

## 発生回数

1回目（今回の P6 ドッグフーディングで初めて発生。**2ストライクルール未該当のため、
hook や CI での強制への昇格はここでは提案しない**）

## 影響

- `gate` イベントが実質的に記録されない運用になり得る。`gate` イベントは差し戻し率・肥大率の
  一部（`kind` 別カウント）の情報源であり、これが空だと集計器のKPIの一部が痩せる。
- ただし `completed`・`submitted`・`issued` は自動記録されているため、**北極星（AFK完走率）
  自体は生存する**。今回の観察は北極星をブロックしない中程度の欠落。

## 提案（強制力の階段を1段だけ——今回は「宣言」まで）

- **現在**：宣言なし。
- **提案**：`docs/playbook.md`「マージ（不可逆操作）」節に一文追記する。

  > 統合役は、マージする前に対象スライスの `gate` イベント（`/gate GO` または
  > `/gate NO-GO --kind ...`）が記録されていることを確認する。無ければ先に PM へ `/gate` の
  > 投稿を依頼する。

- 事後検証（CI で「merge 済みなのに `gate` イベントが無い」を検知する）や実行時強制
  （ブランチ保護の required check 化）への昇格は、**2回目が発生してから**検討する
  （2ストライクルール）。今回は宣言のみに留めるのが妥当と考える。

## 判断待ちの点

- 上記の一文を `docs/playbook.md` に追記してよいか（PM/Harness-Keeper 判断）。
- `gate` イベント欠落を集計器がどう扱うか（現状 `compute_kpis()` は `gate` イベントが無くても
  静かに動く。欠落を明示的にKPIへ反映する設計にするかは別途検討）。
